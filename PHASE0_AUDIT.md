# Phase 0 Audit — n8n touch-points

Complete inventory of every file, line, and concept referencing `n8n`, `N8N_SECRET`, `x-n8n-secret`, or `/api/internal/*`. Phase 1 consumes this list to know exactly what to delete, rewrite, or extract.

Generated by scanning the repo on 2026-04-14.

---

## 1. Code that must be deleted

### 1.1 `lib/n8n-auth.ts` — DELETE entire file
The only helper: reads `process.env.N8N_SECRET`, compares to `x-n8n-secret` header. Nothing else imports it outside the files listed below.

### 1.2 `app/api/internal/` — DELETE entire directory (4 routes)

| Route | File | Method | What it does |
|---|---|---|---|
| `/api/internal/user-register` | `app/api/internal/user-register/route.ts` | POST | Upserts a user by `telegramId`. Called by the `/start` workflow. |
| `/api/internal/task-summary` | `app/api/internal/task-summary/route.ts` | POST | Returns open/overdue/today counts for a `telegramId`. Called by `/tasks` command + daily digest. |
| `/api/internal/due-reminders` | `app/api/internal/due-reminders/route.ts` | GET | Lists tasks due in the next 24 h with a valid assignee `telegramId`. Called by the reminder cron. |
| `/api/internal/notifications` | `app/api/internal/notifications/route.ts` | POST | Logs an `ActivityLog` entry with `action: "notification_sent"`. Called by every outbound-notification workflow. |

All four start with `if (!validateN8nSecret(req)) return 401;`.

**Phase 1.3 extraction targets** — the business logic from each route should move into `lib/` as plain functions:
- `user-register` → `lib/user-upsert.ts` → `upsertUserByTelegramId(...)`
- `task-summary` → `lib/task-summary.ts` → `getTaskSummaryForTelegramUser(telegramId)`
- `due-reminders` → `lib/due-reminders.ts` → `getDueRemindersNext24h()`
- `notifications` → `lib/activity.ts` → `logNotificationSent({ taskId, userId, type })`

### 1.3 `n8n-workflows/` — DELETE entire directory

| File | Trigger | Calls |
|---|---|---|
| `01-start-command.json` | Telegram `/start` | `POST /api/internal/user-register` + sends welcome DM |
| `02-tasks-command.json` | Telegram `/tasks` | `POST /api/internal/task-summary` + sends summary DM |
| `03-task-assigned-notification.json` | Webhook from app (on task create/assign) | Sends DM + `POST /api/internal/notifications` |
| `04-due-reminders.json` | Schedule (every N minutes) | `GET /api/internal/due-reminders` → DM each → `POST /api/internal/notifications` |
| `05-daily-digest.json` | Schedule (daily, 08:00) | `GET /api/users` (with n8n header) → `POST /api/internal/task-summary` per user → DM |
| `README.md` | — | Docs for importing + env vars |

- **Lines 79–94:** The `GET` handler has a dual-path design: if `x-n8n-secret` is valid it returns **all** users (including `telegramId`); otherwise it falls through to the JWT-guarded `getUsersWithJwt`. Strip the n8n branch; the route becomes JWT-only.
- **Line 79 comment:** `/** JWT (app) or 'x-n8n-secret' (n8n digest / automation). */` — rewrite.
- After changes, the function body collapses to: `await connectDB(); return getUsersWithJwt(req, ctx ?? {});`
